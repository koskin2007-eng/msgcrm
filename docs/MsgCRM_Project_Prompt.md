# Проект: MsgCRM

Нужно разработать первый веб-интерфейс MVP для проекта **MsgCRM**.

**MsgCRM** — это единое окно/CRM для продавцов, которые получают сообщения из разных аккаунтов Авито, а в будущем из Дрома, Юлы, VK, Telegram и других каналов.

Главная идея проекта:

> Продавец или компания заходит в свой аккаунт MsgCRM, подключает несколько внешних аккаунтов Авито и других площадок, получает все сообщения в одном интерфейсе и может отвечать клиентам прямо из MsgCRM.

---

# 1. Важное разделение аккаунтов

Нужно различать два типа аккаунтов.

## 1.1. Аккаунт клиента MsgCRM

Это пользователь, который регистрируется в системе MsgCRM, входит по email/паролю и работает внутри своей компании.

## 1.2. Подключённые внешние аккаунты

Это аккаунты Авито, Дром, Юла, VK, Telegram и других площадок, которые клиент подключает внутри своего кабинета MsgCRM.

Каждый клиент MsgCRM должен иметь своё отдельное рабочее пространство:

- Company;
- Workspace.

Один Workspace не должен видеть данные другого Workspace.

Пример:

Компания **“АвтоПлюс”** видит только свои диалоги, свои объявления, своих сотрудников и свои подключённые аккаунты.

Компания **“Недвижимость 24”** видит только свои данные.

На этом этапе реальную авторизацию и реальные API подключать не нужно.

Нужно сделать frontend-прототип с mock-данными, но структура должна быть подготовлена под будущую backend-логику.

- красный — ошибки авторизации;
- серый — каналы “скоро” или неактивные элементы.

# 19. Поведение в MVP

Нужно сделать кликабельный прототип.

Минимальная логика:

- при выборе диалога в списке открывается соответствующий чат;
- кнопки быстрых ответов могут подставлять текст в поле ответа;
- кнопка “Отправить” добавляет mock-сообщение в чат;
- кнопка “Создать сделку” может менять статус диалога или создавать mock-сделку;
- фильтры могут работать на mock-данных;
- переходы между страницами должны работать.

Реальное API пока не подключать.

Но структура должна быть готова к будущему подключению backend.

---

# 20. Что не делать в первой версии

Не делать в MVP:

- реальную интеграцию с Авито API;
- реальную регистрацию через backend;
- оплату;
- сложную аналитику;
- складской учёт;
- телефонию;
- мобильное приложение;
- сложную систему прав;
- реальный расчёт доставки через API.

Сначала нужен понятный рабочий интерфейс:

- вход;
- компания;
- команда;
- интеграции;
- входящие сообщения;
- чат;
- быстрые ответы;
- карточка объявления;
- карточка клиента;
- начало сделки.

---

# 21. Главная цель

Главная цель — получить первый кликабельный интерфейс MVP, который можно показать как основу продукта MsgCRM.

MsgCRM должен визуально объяснять идею:

> Все сообщения продавца из Авито и других каналов — в одном окне. Отвечайте быстрее, не теряйте клиентов и доводите переписку до сделки.

Сделай интерфейс простым, чистым и понятным.
